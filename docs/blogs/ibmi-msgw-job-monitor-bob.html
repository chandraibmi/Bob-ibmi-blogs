From a Single Sentence Requirement to a Production-Ready RPGLE Application

The Business Requirement

In many IBM i environments, production job monitoring follows a well-established operational process. Experienced administrators routinely use WRKACTJOB, review job logs, and respond to system messages as part of their daily responsibilities. For teams with deep platform knowledge, this approach works effectively.

The problem was our Level 1 (L1) Service Desk team. L1 analysts handle first-line production monitoring during off-hours and peak periods, but they do not have the IBM i command skills that operations staff do. When a batch job stalls in MSGW (Message Wait) status — waiting for an operator to respond to a system inquiry message — the L1 team had no straightforward way to identify or act on it. Asking them to run WRKACTJOB and filter through hundreds of active jobs was not realistic.

As an IBM i developer, I saw a clear opportunity: build a simple, focused interface that does the filtering for them. A screen that shows only the MSGW jobs, with labelled action options right there — no command knowledge needed. The L1 analyst opens the screen, sees what needs attention, and takes action directly.

That was the requirement. I used Bob, an AI-powered IBM i development assistant, to build it — and documented the full journey here.

Generating the RPGLE Program

The SQLRPGLE program (MSGWMON) is fully free-format (**free), thoroughly commented, and organized into focused subroutines.

The QCMDEXC Prototype

RPGLE — QCMDEXC Prototype 
// IMPORTANT: Cmd must be declared as char OPTIONS(*VARSIZE), NOT varchar.
// Declaring Cmd as varchar causes RPG to include the internal 2-byte length
// prefix in the data stream. QCMDEXC then interprets those two bytes as the
// first characters of the command string, producing an 'invalid character'
// error. The IBM-documented prototype uses char.
dcl-pr QCMDEXC extpgm('QCMDEXC');
  Cmd     char(32702) options(*varsize) const;
  CmdLen  packed(15:5)                 const;
end-pr;

The SQL Cursor

Embedded SQL — Cursor Declaration 
exec sql
  DECLARE C_MSGWJOBS CURSOR FOR
    SELECT
      -- JOB_NAME from ACTIVE_JOB_INFO is already 'number/user/name' (28 chars)
      JOB_NAME,
      -- Extract the short job name: everything after the second slash
      SUBSTR(JOB_NAME,
             LOCATE('/', JOB_NAME, LOCATE('/', JOB_NAME) + 1) + 1)
        AS JOB_NAME_SHORT,
      JOB_USER,
      JOB_NUMBER,
      COALESCE(JOB_TYPE,      '   '),
      COALESCE(SUBSYSTEM,     '          '),
      COALESCE(FUNCTION_TYPE, '  '),
      COALESCE(FUNCTION,      '          ')
    FROM TABLE(QSYS2.ACTIVE_JOB_INFO())
    WHERE JOB_STATUS = 'MSGW'
    ORDER BY JOB_TYPE, SUBSYSTEM, JOB_USER, JOB_NAME;

A critical nuance Bob handled correctly: JOB_NAME in ACTIVE_JOB_INFO is already the fully qualified name (number/user/name). The nested LOCATE + SUBSTR expression extracts only the short job name for the display column, while the full qualified name is stored in SFQJOB for command execution.

Subfile Load Logic

RPGLE — LoadSubfile Subroutine 
begsr LoadSubfile;

  SflDsp    = *off;
  SflDspCtl = *off;
  SflClr    = *on;
  SflEnd    = *off;
  CntGrn    = *off;
  CTJOBCNT  = 0;
  write MSGCTL;       // Clear the subfile
  SflClr    = *off;

  WkJobCnt = 0;
  WkRRN    = 0;

  exec sql OPEN C_MSGWJOBS;

  exec sql
    FETCH NEXT FROM C_MSGWJOBS
    INTO :JobRow.JobName,  :JobRow.JobShortName,
         :JobRow.JobUser,  :JobRow.JobNumber,
         :JobRow.JobType,  :JobRow.Subsystem,
         :JobRow.FuncType, :JobRow.Function;

  dow SQLCODE = 0;
    WkJobCnt += 1;
    WkRRN    += 1;

    SFOPT      = '   ';
    SFJOBNUM   = %char(JobRow.JobNumber);
    SFJOBUSR   = %char(JobRow.JobUser);
    SFJOBNAM   = %char(JobRow.JobShortName);   // Short name for display
    SFJOBTYP   = %char(JobRow.JobType);
    SFSUBSYS   = %char(JobRow.Subsystem);
    SFTYPE     = %char(JobRow.FuncType);
    SFFUNCTION = %char(JobRow.Function);
    SFQJOB     = %char(JobRow.JobName);        // Full qualified name (hidden)

    write MSGSFL;

    exec sql FETCH NEXT FROM C_MSGWJOBS
      INTO :JobRow.JobName,  :JobRow.JobShortName,
           :JobRow.JobUser,  :JobRow.JobNumber,
           :JobRow.JobType,  :JobRow.Subsystem,
           :JobRow.FuncType, :JobRow.Function;
  enddo;

  exec sql CLOSE C_MSGWJOBS;

  CTJOBCNT  = WkJobCnt;
  CntGrn    = (WkJobCnt > 0);
  SflDsp    = (WkJobCnt > 0);
  SflDspCtl = *on;
  SflEnd    = *on;

endsr;

Processing Operator Options

RPGLE — ProcessOptions Subroutine 
begsr ProcessOptions;

  WkEof = *off;
  readc MSGSFL;
  WkEof = %eof(MSGWDSPF);

  dow not WkEof;

    WkQualJob = %trimr(SFQJOB);   // varchar — no trailing space issues

    select;

      when %trim(SFOPT) = '2';    // Work With Job
        WkCmd    = 'WRKJOB JOB(' + WkQualJob + ')';
        WkCmdLen = %len(WkCmd);
        QCMDEXC(WkCmd: WkCmdLen);

      when %trim(SFOPT) = '3';    // Hold Job
        WkCmd    = 'HLDJOB JOB(' + WkQualJob + ') DUPJOBOPT(*MSG)';
        WkCmdLen = %len(WkCmd);
        monitor;
          QCMDEXC(WkCmd: WkCmdLen);
          WkMsgText = 'Hold submitted: ' + WkQualJob;
          exsr SendMsgText;
        on-error;
          WkMsgText = 'Hold failed: ' + WkQualJob + '. Check job log.';
          exsr SendMsgText;
        endmon;

      when %trim(SFOPT) = '4';    // End Job — Controlled
        WkCmd    = 'ENDJOB JOB(' + WkQualJob +
                   ') OPTION(*CNTRLD) DELAY(30) DUPJOBOPT(*MSG)';
        WkCmdLen = %len(WkCmd);
        monitor;
          QCMDEXC(WkCmd: WkCmdLen);
          WkMsgText = 'End (Ctrl) submitted: ' + WkQualJob;
          exsr SendMsgText;
        on-error;
          WkMsgText = 'End (Ctrl) failed: ' + WkQualJob + '. Check job log.';
          exsr SendMsgText;
        endmon;

      when %trim(SFOPT) = '5';    // Display Messages
        WkCmd    = 'WRKJOB JOB(' + WkQualJob + ') OPTION(*MSG)';
        WkCmdLen = %len(WkCmd);
        QCMDEXC(WkCmd: WkCmdLen);

      when %trim(SFOPT) = '6';    // Release Job
        WkCmd    = 'RLSJOB JOB(' + WkQualJob + ') DUPJOBOPT(*MSG)';
        WkCmdLen = %len(WkCmd);
        monitor;
          QCMDEXC(WkCmd: WkCmdLen);
          WkMsgText = 'Release submitted: ' + WkQualJob;
          exsr SendMsgText;
        on-error;
          WkMsgText = 'Release failed: ' + WkQualJob + '. Check job log.';
          exsr SendMsgText;
        endmon;

      when %trim(SFOPT) = '7';    // Display Job Log
        WkCmd    = 'DSPJOBLOG JOB(' + WkQualJob + ') OUTPUT(*)';
        WkCmdLen = %len(WkCmd);
        QCMDEXC(WkCmd: WkCmdLen);

      when %trim(SFOPT) = '8';    // End Job — Immediate
        WkCmd    = 'ENDJOB JOB(' + WkQualJob +
                   ') OPTION(*IMMED) DUPJOBOPT(*MSG)';
        WkCmdLen = %len(WkCmd);
        monitor;
          QCMDEXC(WkCmd: WkCmdLen);
          WkMsgText = 'End (Immed) submitted: ' + WkQualJob;
          exsr SendMsgText;
        on-error;
          WkMsgText = 'End (Immed) failed: ' + WkQualJob + '. Check job log.';
          exsr SendMsgText;
        endmon;

      other;
        WkMsgText = 'Option "' + %trimr(SFOPT) + '" not valid. Use 2-8.';
        exsr SendMsgText;

    endsl;

    SFOPT = '   ';
    update MSGSFL;

    readc MSGSFL;
    WkEof = %eof(MSGWDSPF);

  enddo;

endsr;

Key observation: This is one of the most underrated qualities of AI-assisted development — surgical precision on incremental changes. Every enhancement touched only what needed to change.

The Runtime Bug — and How Bob Diagnosed It

After the enhancement recompile, I discovered a behavior defect during testing.

Symptom: Every time the subfile refreshed — pressing F5 or returning from an action — the function key legend on lines 22–23 was wiped off the screen.

I reported it to Bob:

"The function key footer disappears every time the subfile refreshes."

Bob's diagnosis was immediate, precise, and technically correct:

"The root cause is that EXFMT MSGCTL performs a full-screen write operation. Without OVERLAY on the subfile control record, the 5250 data stream clears all screen positions not explicitly written by that operation — including the FOOTER record on lines 22–23. Adding OVERLAY to MSGCTL instructs the 5250 write operation to merge its output with the existing screen content rather than replacing it entirely."

❌ Before — Missing OVERLAY
     A          R MSGCTL                    SFLCTL(MSGSFL)
     A                                      SFLSIZ(9999)
     A                                      SFLPAG(0015)
✅ After — OVERLAY Added
     A          R MSGCTL                    SFLCTL(MSGSFL)
     A                                      OVERLAY
     A                                      SFLSIZ(9999)
     A                                      SFLPAG(0015)

One DDS keyword. Bob identified it, explained the underlying 5250 write semantics, updated the source, and the issue was resolved after recompilation. The FOOTER remained persistent on every subsequent refresh.

Other Subtle IBM i Bugs Bob Caught

The development journey was not perfectly linear. Several IBM i-specific defects surfaced during the iterative testing cycle. Bob identified and resolved all of them.

Bug 1 — Corrupt Qualified Job Name

Symptom: All CL commands (HLDJOB, ENDJOB, etc.) were failing with "job not found" errors.

Root cause: Early code was reconstructing the qualified job name by concatenating JobNumber + '/' + JobUser + '/' + JobName. What was not immediately obvious is that JOB_NAME from ACTIVE_JOB_INFO is already the fully qualified name in number/user/name format. The reconstruction was double-encoding it, producing corrupt values like 327961/BOB4IPGMR/327961/BOB4 (truncated at 28 characters).

❌ Incorrect
// JOB_NAME is already the full qualified name
SFQJOB = %char(JobRow.JobNumber) + '/' +
         %char(JobRow.JobUser)   + '/' +
         %char(JobRow.JobName);
✅ Correct
// Assign directly — no reconstruction needed
SFQJOB   = %char(JobRow.JobName);
SFJOBNAM = %char(JobRow.JobShortName);

Bug 2 — QCMDEXC varchar vs char Parameter Type

Symptom: Every command via QCMDEXC failed with "contains a character that is not valid" — even syntactically correct commands.

Root cause: When a varchar variable is passed to a varchar OPTIONS(*VARSIZE) parameter in ILE RPG, the compiler includes the internal 2-byte length prefix as part of the data stream. QCMDEXC interprets those two bytes as the first two characters of the command string, producing '  DSPJOBLOG JOB(...)' — two invisible leading characters that make the command invalid.

❌ Incorrect — varchar
dcl-pr QCMDEXC extpgm('QCMDEXC');
  Cmd     varchar(32702)
           options(*varsize) const;
  CmdLen  packed(15:5) const;
end-pr;
✅ Correct — char
dcl-pr QCMDEXC extpgm('QCMDEXC');
  Cmd     char(32702)
           options(*varsize) const;
  CmdLen  packed(15:5) const;
end-pr;
Why this matters: This is a genuinely obscure IBM i calling-convention nuance. It is documented, but it sits in a corner of the ILE RPG Programmer's Guide that most developers encounter only after hours of debugging a symptom that looks completely unrelated to a prototype declaration.

Bug 3 — WRKMSG Is Not a Valid Command for Jobs

Symptom: Option 5 (Display Messages) failed with "command not found" or "parameter not valid" errors.

Root cause: The initial implementation used WRKMSG JOB(...). The WRKMSG command operates on message queue objects (e.g., WRKMSG MSGQ(QGPL/QSYSOPR)), not job names. It does not accept a JOB() parameter.

❌ Incorrect
WkCmd = 'WRKMSG JOB(' + WkQualJob + ')';
✅ Correct
WkCmd = 'WRKJOB JOB(' + WkQualJob + ')
         OPTION(*MSG)';
